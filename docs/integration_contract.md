# Integration contract: требования fl-web-service к flwr-репо

## Что это за документ и как с ним работать

Это **односторонние требования** со стороны веб-сервиса (`fl-web-service`) к
ML-репозиторию (`flwr-репо`, рабочее имя `ddl`). Документ не описывает,
что в flwr-репо **уже есть**, — он описывает, что веб-сервису **нужно**,
чтобы выполнить свою часть дипломного проекта по дорожной карте.

Рабочий процесс:

1. Веб-сервис (этот репо) поддерживает этот файл как источник истины по
   своим нуждам.
2. Изменения в этом файле сначала отправляются на сторону flwr-репо.
3. Сторона flwr-репо отвечает, что из этого уже есть, что добавит, а что
   требует переформулировки. Ответ возвращается в этот репо.
4. На основе ответа в этом файле появляется раздел «Реализация» с фактически
   доступными именами модулей, функций, форматами событий и т. д.

Пока раздела «Реализация» нет — ниже только требования и открытые вопросы.

## Контекст

- **Цель веб-сервиса:** регистрировать участников, упаковать FL-клиента в
  Docker-образ, позволить запускать FL-эксперименты через веб-интерфейс,
  показывать live-метрики раундов, хранить обученные модели и обслуживать
  инференс.
- **Уже работает в flwr-репо (по словам автора):** SuperLink на приватной
  облачной VM, 10 SuperNode'ов (7 на приватном облаке + 3 на Yandex Cloud),
  обучение на CIFAR-100, текущие эксперименты по system- и statistical-
  heterogeneity.
- **Роли сторон:** flwr-репо — всё ML-ядро (модели, данные, стратегии,
  тренировочные циклы, партиционирование). fl-web-service — веб, БД, Docker-
  образ клиента, оркестрация SuperLink, реестр, инференс. Связь между репо
  **односторонняя**: fl-web-service импортирует из flwr-репо, обратно — нет.
- **Ограничения:** CPU-only на всех узлах. Срок спринта — 3 недели фуллтайм,
  старт 2026-04-11. Два датасета: CIFAR-100 (отладка) и PlantVillage (демо).

## Что веб-сервису нужно от flwr-репо

### 1. Публичный Python-API для оркестрации SuperLink

Веб-сервис хочет уметь из своего backend-процесса запустить Flower SuperLink,
указав ему стратегию и её параметры, без дублирования ML-кода.

Требования:

- **R1.1.** Должен существовать способ получить объект стратегии Flower
  (`flwr.server.Strategy` или его наследник) по её строковому имени и словарю
  параметров. Имена и допустимые параметры определяются и документируются в
  flwr-репо.
- **R1.2.** Должен существовать способ запустить SuperLink программно
  **или** как документированную CLI-команду. Допустимы оба варианта; если
  программный API отсутствует, веб-сервис запустит задокументированную
  команду через `subprocess`.
- **R1.3.** SuperLink должен быть сконфигурируем по:
  - адресу и портам (fleet, exec — сейчас это 9092 и 9093);
  - числу раундов;
  - минимальному числу клиентов для старта раунда;
  - имени стратегии и её параметрам (см. R1.1);
  - имени датасета, с которым идёт эксперимент (сейчас — `cifar100`, позже —
    `plantvillage`).
- **R1.4.** SuperLink должен корректно завершаться по сигналу
  (`SIGTERM` → graceful shutdown в разумный срок). Веб-сервис будет
  останавливать его как subprocess.

### 2. Публичный Python-API для клиентской части

Веб-сервис собирает Docker-образ `fl-client`, внутри которого лежит код
клиента из flwr-репо. Клиент на целевой VM запускает этот контейнер, передаёт
в него адрес SuperLink и свой идентификатор через переменные окружения,
и становится полноценным SuperNode'ом.

Требования:

- **R2.1.** Должна существовать точка входа для клиента: либо функция
  `client_fn(...) -> flwr.client.Client`, либо готовый `ClientApp`, либо
  CLI-команда, которую можно вызвать из entrypoint Docker-контейнера.
- **R2.2.** Клиент должен принимать конфигурацию как минимум из следующих
  источников:
  - адрес SuperLink (переменная окружения);
  - идентификатор клиента (переменная окружения, выдаётся веб-сервисом);
  - локальный путь к данным (переменная окружения);
  - имя датасета (переменная окружения или параметр в config).
- **R2.3.** Клиент должен уметь работать со **обоими** датасетами: CIFAR-100
  и PlantVillage. Переключение — через конфиг, без пересборки образа.
- **R2.4.** Клиент должен работать **только на CPU**. GPU-сборки не нужны
  ни на одной стороне.
- **R2.5.** Клиент должен корректно завершаться между раундами, не падая при
  временной потере связи с SuperLink в течение разумного таймаута.

### 3. Модели и датасеты

- **R3.1.** Для каждого датасета в flwr-репо должна быть документирована
  выбранная модель и её размер в параметрах. Модели должны быть CPU-friendly
  (в идеале ≤ 1M параметров для PlantVillage; для CIFAR-100 — чем меньше,
  тем лучше для скорости итераций).
- **R3.2.** Для PlantVillage должна быть документирована схема партиционирования
  данных: как клиенты получают свои куски данных (локальные файлы, URL
  скачивания, пресет в MinIO — любой вариант, главное документированный).
- **R3.3.** Должен быть способ получить число классов и список имён классов
  (нужно для инференса и UI). Форма поставки — константа, функция или
  JSON-файл в репо.
- **R3.4.** Финальные веса обученной модели должны быть сохраняемы в файл
  (`state_dict` PyTorch или эквивалент), который веб-сервис положит в MinIO
  и потом загрузит обратно в inference-worker.

### 4. События раундов для live-дашборда

Веб-сервис показывает живой дашборд: прогресс текущего раунда, список активных
клиентов, графики loss/accuracy. Ему нужны события, которые он сможет
ретранслировать в UI через WebSocket.

- **R4.1.** Стратегия (или обучающий цикл SuperLink) должна уметь публиковать
  события по ходу обучения:
  - `round_started` (номер раунда, время, число подключённых клиентов);
  - `round_finished` (номер раунда, агрегированные метрики, длительность);
  - `client_update` (идентификатор клиента, номер раунда, локальные метрики,
    число образцов).
- **R4.2.** Канал публикации — Redis pub/sub. Redis будет жить на стороне
  веб-сервиса; flwr-репо получит URL Redis через переменную окружения
  (`REDIS_URL`).
- **R4.3.** Если встроить публикацию прямо в стратегию тяжело, допустимо
  вместо этого писать события в stdout в строгом JSON-формате (одна строка =
  одно событие). Веб-сервис тогда будет парсить stdout subprocess'а. Это
  запасной вариант; первичный — Redis.
- **R4.4.** Формат события — JSON с обязательными полями `type`, `run_id`,
  `ts` (UNIX-время) и специфичной для типа нагрузкой. Финальную схему
  согласуем к Дню 8 дорожной карты.

### 5. Упаковка и версионирование

- **R5.1.** flwr-репо должен быть **устанавливаем как Python-пакет**. Любой из
  вариантов подходит:
  - publishable package с именем в `pyproject.toml` (тогда
    `pip install git+https://...` работает);
  - editable install из локального пути на машине сборки
    (`pip install -e /path/to/flwr-repo`);
  - копирование исходников внутрь Docker-образа клиента на этапе сборки.

  Нам нужна **одна из этих опций**, работающая и задокументированная. Какая
  именно — выберет сторона flwr-репо.
- **R5.2.** Версия `flwr` на стороне flwr-репо пинится в его `pyproject.toml`
  или `requirements.txt`. Веб-сервис поставит **ту же самую версию** у себя
  и в Docker-образе клиента. Нужно знать точную версию.
- **R5.3.** Версия Python на стороне flwr-репо — **3.12** (подтверждено).
  Веб-сервис работает на той же версии.
- **R5.4.** Все ML-зависимости (`torch`, датасеты, и т.п.) ставятся через
  установку пакета flwr-репо. Веб-сервис не дублирует их в своём
  `pyproject.toml`.

### 6. То, что веб-сервису НЕ нужно

Явно фиксирую, чтобы не было лишней работы на стороне flwr-репо:

- **Не нужно** автоматически управлять SSH-деплоем клиентов на VM через
  `nodes.conf`. Это остаётся внутренним инструментом flwr-репо и
  веб-сервисом не используется.
- **Не нужно** шифрование каналов и TLS между SuperLink и SuperNode'ами.
  Это учебный проект, никаких реальных атакующих нет.
- **Не нужна** аутентификация клиентов в протоколе Flower. Идентификация
  клиентов делается на стороне веб-сервиса через opaque-токен как чисто
  UX-метка на дашборде.
- **Не нужна** дифференциальная приватность, secure aggregation — они вынесены
  в «направления дальнейшего развития» в тексте диплома.
- **Не нужен** GPU-путь ни для сервера, ни для клиента.

## Открытые вопросы к стороне flwr-репо

Эти вопросы веб-сервис задаёт и ждёт ответа. На основе ответов будет написан
раздел «Реализация» ниже.

1. **Имя пакета.** Какое имя у Python-пакета flwr-репо в его `pyproject.toml`?
   (Рабочая гипотеза — `ddl` по имени директории, но это нужно подтвердить.)
2. **Версия flwr.** Какая точная версия `flwr` используется сейчас на сервере
   и на клиентах? (`pip show flwr` в активном venv.)
3. **Версия torch и других крупных зависимостей.** Что стоит в `pyproject.toml`
   или `requirements.txt` flwr-репо?
4. **Способ установки flwr-репо как зависимости.** Какой из трёх вариантов из
   R5.1 предпочтителен на стороне flwr-репо?
5. **API оркестрации SuperLink.** Есть ли программный Python-API для запуска
   SuperLink со стратегией и параметрами, или только CLI-команда? Если CLI —
   какая именно команда и какие у неё аргументы?
6. **Точка входа клиента.** Что именно должен вызывать entrypoint Docker-
   контейнера: `python -m <module>`, готовый `flwr client …`, или что-то
   другое? Какие переменные окружения клиент уже читает сейчас?
7. **Список стратегий.** Какие стратегии агрегации сейчас доступны в
   flwr-репо и какие у них настраиваемые параметры?
8. **Список моделей.** Какая модель используется сейчас для CIFAR-100? Какая
   планируется для PlantVillage? Сколько в ней параметров?
9. **Партиционирование данных.** Как сейчас клиенты получают свои куски
   CIFAR-100? Как планируется делать то же самое для PlantVillage?
10. **Сохранение весов.** Где и как сейчас сохраняются финальные веса после
    завершения обучения? Есть ли функция, которая принимает путь и пишет в
    него `state_dict`?
11. **События раундов.** Есть ли уже какие-то callback'и в стратегии,
    которые можно расширить публикацией в Redis? Или стратегия «глухая»
    и события придётся добавлять с нуля?
12. **Число классов и их имена.** Как сейчас организованы метаданные датасета
    (число классов, список имён)? Откуда веб-сервис сможет их получить для
    UI инференса?
